# 文心管理 App PRD

## 1. 产品概述

文心管理 App 是为餐饮品牌「文心」设计的内部运营管理系统。产品目标是让老板、管理层和各岗位员工在同一个系统中完成日常经营查看、审批处理、Bento 订单协作、采购、员工管理、财务查看和门店运营管理。

当前阶段优先设计老板视角 App 首页，以及后续可进入的堂食、Bento、审批、报表和功能入口页面。

产品核心原则：

- 首页只显示老板需要立刻知道的结论，不展示复杂过程。
- 详细数据放入下级页面，点击后再查看。
- 支持多角色权限、多语言、多店铺，为未来扩展做基础。
- 现阶段不直接重做 POS，优先考虑接入现有 POS 数据。

## 2. 目标用户与角色

### 2.1 用户角色

| 角色 | 权限范围 |
| --- | --- |
| 老板 Bruce | 全权限，可查看全部店铺、全部数据、审批、报表和系统设置 |
| 管理层 | 菜单、采购审批、工资审核、违规记录、报表只读 |
| 后厨 | 今日菜品状态、备料清单、Bento 制作清单 |
| 打包 | Bento 打包清单、订单备注、打包确认 |
| 配送 | 配送地址、客户联系、送达确认 |
| 前厅员工 | 菜单只读、自己的考勤与违规 |
| 收银 / 管理层 | 订单管理、日结报表 |

### 2.2 角色设计原则

- 每个账号必须绑定角色。
- 每个账号必须绑定所属店铺。
- 老板可查看全部店铺或切换单店视角。
- 员工默认只能看到自己岗位需要的功能。
- 所有菜单、报表、审批入口按角色动态显示。

## 3. 多语言需求

系统需要支持至少三种语言：

- 中文
- Bahasa Melayu
- English

### 3.1 使用场景

- 老板 Bruce 默认使用中文。
- 马来员工默认使用 Bahasa Melayu。
- 管理层可选择中文或 English。
- 员工手册、SOP、违规记录、请假规则等内容需要支持多语言展示。

### 3.2 功能要求

- 每个账号保存自己的语言偏好。
- 在「我的 → 设置 → 语言」中切换语言。
- 系统文案不能写死在页面中，需要通过语言包管理。
- 后续员工手册、SOP、通知模板应支持三语版本。

## 4. 多店铺需求

系统需要支持一个品牌下多家门店。

### 4.1 店铺切换

示例：

总店 · 5月20日 周一 · 营业中 ▼

点击后可选择：

- 全部店铺
- 总店
- 二店
- 三店
- Bento Central Kitchen

### 4.2 视图模式

- 总异常数
- 总待处理数
- 各店营业状态

#### 单店模式

显示：

- 该店今日营业额
- 该店异常
- 该店待处理事项
- 该店堂食情况
- 该店 Bento 情况

### 4.3 数据要求

以下核心数据都需要带 `store_id`：

- 员工
- 菜单
- 订单
- Bento 订单
- 采购
- 排班
- 考勤
- 违规记录
- 报表
- 审批
- 当日事项

## 5. 当前首页设计方向

### 5.1 首页定位

老板首页不是报表页，也不是功能大全页。首页应是老板打开 App 后的一眼总览。

老板打开 App 后，3 秒内应该知道：

1. 今天总营业额是多少。
2. 当前是否有异常。
3. 当前是否有待处理事项。
4. 堂食是否正常。
5. Bento 是否正常。
6. 今天店内有什么重点事项。

### 5.2 首页设计原则

- 信息少而准。
- 字体要大，适合手机查看。
- 不放复杂图表。
- 不放趋势线。
- 不放大面积装饰插画。
- 不把详情页信息塞进首页。
- 首页只展示总结，点击后进入下级页面。

## 6. 老板 App 首页内容结构

### 6.1 顶部 Header

包含：

- 菜单图标
- 文心 Logo
- 标题：文心管理系统
- 通知铃铛，显示未读数量
- Bruce 头像
- Bruce 名字
- 下拉箭头

要求：

- Header 只作为身份区，不作为主视觉。
- Logo 和头像不能过大。
- 通知红点保留，但不能太抢眼。

### 6.2 问候与营业状态

文案示例：

```text
早上好，Bruce
5月20日 周一 · ● 营业中
```

要求：

- 营业状态使用绿色圆点或绿色文字。
- 不使用大营业状态卡。
- 店铺选择器可合并在这一行，例如「总店 · 5月20日 周一 · 营业中 ▼」。

### 6.3 今日营业额

这是首页最重要的信息。

内容：

```text
今日营业额                 报表 >
RM 2,835
堂食 + Bento · 较昨日 +10.8%
```

要求：

- `RM 2,835` 是页面最大数字。
- 删除营业额卡中的店铺插画。
- 删除大按钮样式的「查看报表」。
- 右上角只保留小文字入口「报表 >」。
- 不展示堂食营业额、Bento 营业额、成本率等详细分项。
- 分项数据进入报表页或堂食 / Bento 详情页查看。

### 6.4 今日状态一览

显示三个简洁状态块：

```text
异常
4 项

待处理
12 项

Bento
进行中
```

点击逻辑：

- 点击「异常」进入异常详情页，查看迟到、旷工、报修、紧急补货、库存异常等。
- 点击「待处理」进入待处理事项页，查看审批、请假、采购、设备报修等。
- 点击「Bento」进入 Bento 今日进度页。

视觉要求：

- 三张卡格式统一。
- 图标要小，不抢数字。
- 异常用红 / 浅红。
- 待处理用橙 / 浅橙。
- Bento 正常用绿 / 浅绿。

### 6.5 业务概况

显示两个入口卡：

#### 堂食

```text
堂食
正常营业
RM 1,850
```

点击进入堂食详情页。

#### Bento

```text
Bento
进行中
128 单 · 完成 74%
```

点击进入 Bento 详情页。

要求：

- 两张卡只显示核心概况。
- 不展示趋势线。
- 不展示复杂图表。
- 图标缩小，减少装饰感。
- 箭头保留，表示可进入详情。

### 6.6 当日事项

标题：

```text
当日事项                         全部 >
```

首页显示 3 条最重要事项：

```text
今日主推：烤鱼
今日例汤：冬瓜排骨汤
15:00 员工卫生培训
```

要求：

- 首页只显示重点事项。
- 不使用复杂表格。
- 不使用插画。
- 可点击「全部」进入完整当日事项列表。

### 6.7 功能入口按钮

首页不直接展示所有快捷入口。

设计：

- 右下角放一个悬浮九宫格按钮。
- 点击后弹出功能启动台。
- 功能启动台类似 Mac Launchpad。
- 推荐从底部弹出全屏或 80% 高度面板。

功能启动台包含：

- 运营管理
- 采购管理
- 员工管理
- 财务管理
- Bento 管理
- 审批中心
- 排班表
- 考勤打卡
- 员工手册
- 系统设置

### 6.8 底部导航

底部导航 5 个：

```text
首页 / 审批 / Bento / 报表 / 我的
```

要求：

- 当前选中「首页」，使用暖橙色。
- 其他图标使用统一灰色。
- 审批可显示待审批红点，但需要与顶部通知红点区分。

## 7. 堂食详情页需求

堂食详情页用于查看当前餐厅堂食运营情况。

### 7.1 核心信息

应显示：

- 当前营业状态
- 今日堂食营业额
- 当前就餐桌数
- 空桌数
- 已结账未清台数量
- 桌台状态图

### 7.2 桌台信息

每张桌显示：

- 桌号
- 当前状态：空桌 / 就餐中 / 已结账 / 待清台
- 人数
- 开台时间
- 当前消费金额

### 7.3 点击桌台后显示

- 点单详情
- 菜品名称
- 数量
- 备注
- 已上菜 / 未上菜状态
- 账单金额
- 支付状态

### 7.4 POS 数据来源

堂食详情页优先从现有 POS 系统读取数据。

需要向 POS 供应商确认是否支持：

- 当前桌台状态 API
- 未结账订单 API
- 订单明细 API
- 菜品、数量、备注 API
- 支付状态 API
- 退款、折扣 API
- 日结报表 API
- Webhook 推送
- 测试环境
- API 是否收费
- 数据延迟是实时、分钟级还是日结后

## 8. POS 系统策略

当前阶段不建议立即完全自研 POS。

推荐策略：

1. 继续使用现有供应商 POS 负责现场点单、收银、打印、退款、日结。
2. 文心管理 App 先做老板管理驾驶舱和餐厅运营中台。
3. 优先接入 POS 数据，实现只读同步。
4. 如果供应商 POS 支持 API，则接入桌台、订单、支付和日结数据。
5. 如果供应商 POS 不支持 API，则先使用 CSV / Excel / 日结报表同步。
6. 等业务流程和数据模型成熟后，再评估是否自研 POS 或更换开放 API 的 POS。

自研 POS 需要谨慎，因为 POS 影响：

- 点单
- 桌台
- 收银
- 打印厨房单
- 钱箱
- 支付
- 退款
- 折扣
- 日结
- 税务
- 断网离线
- 权限控制
