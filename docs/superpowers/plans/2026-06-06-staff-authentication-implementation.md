# Wenxin Staff Authentication Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add secure Staff ID login, first-login password change, four-role authorization, twelve-hour sessions, Owner account administration, and operational audit history to Wenxin App.

**Architecture:** Use Supabase Auth with synthetic internal staff emails and cookie-based SSR through `@supabase/ssr`. Next.js 16 `proxy.ts` performs fast route checks, server utilities enforce authorization near data access, and PostgreSQL RLS remains the final data boundary. Owner-only account actions use a server-only Supabase admin client; operational audit entries are generated by database triggers.

**Tech Stack:** Next.js 16 App Router, React 19, TypeScript, Supabase Auth/Postgres/RLS, `@supabase/ssr`, Node assertion tests, Tailwind CSS.

## Task 4: Create Authentication Schema, Session Enforcement, And RLS

**Files:**
- Create: `supabase/migrations/202606061200_staff_authentication.sql`

- [ ] **Step 1: Write migration assertions**

At the end of the migration transaction, include schema checks that fail deployment when required objects are absent:

```sql
do $$
begin
  if to_regclass('public.staff_profiles') is null
    or to_regclass('public.staff_sessions') is null
    or to_regclass('public.audit_logs') is null then
    raise exception 'staff authentication schema is incomplete';
  end if;
end;
$$;
```

- [ ] **Step 2: Create staff tables**

Create:

```sql
create table public.staff_profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  staff_id text not null unique check (staff_id ~ '^[a-z0-9][a-z0-9._-]{2,31}$'),
  display_name text not null,
  role text not null check (role in ('owner', 'manager', 'kitchen', 'front_desk')),
  active boolean not null default true,
  must_change_password boolean not null default true,
  sessions_invalidated_at timestamptz,
  created_by uuid references auth.users(id),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  last_login_at timestamptz
);

create table public.staff_sessions (
  id uuid primary key,
  staff_user_id uuid not null references public.staff_profiles(id) on delete cascade,
  staff_id text not null,
  started_at timestamptz not null default now(),
  last_seen_at timestamptz not null default now(),
  expires_at timestamptz not null,
  ended_at timestamptz,
  end_reason text check (end_reason in ('logout', 'expired', 'forced', 'suspended')),
  device_summary text not null default ''
);

create table public.audit_logs (
  id bigint generated by default as identity primary key,
  actor_user_id uuid references auth.users(id),
  actor_staff_id text,
  action text not null,
  entity_type text not null,
  entity_id text,
  summary text not null,
  before_data jsonb,
  after_data jsonb,
  created_at timestamptz not null default now()
);
```

- [ ] **Step 3: Create security helpers**

Add security-definer functions with `set search_path = public, auth`:

- `current_session_id()` reads `auth.jwt() ->> 'session_id'`
- `current_staff_role()` reads active `staff_profiles`
- `is_current_staff_session_valid()` verifies active profile, matching `auth.sessions`, under twelve hours, and after `sessions_invalidated_at`
- `start_staff_session(device_summary text)` creates the twelve-hour mirror row
- `touch_staff_session()` updates `last_seen_at` at most once every five minutes
- `end_current_staff_session(reason text)` closes the mirror session
- `complete_first_password_change()` changes only the caller's `must_change_password` flag
- `invalidate_staff_sessions(target_user uuid, reason text)` is Owner-only, updates `sessions_invalidated_at`, closes mirror sessions, and deletes target rows from `auth.sessions`
- `set_bento_order_status(order_id bigint, next_status text)` permits Kitchen to change only fulfillment status

- [ ] **Step 4: Enable RLS on auth and current business tables**

Policies must enforce this matrix:

```text
daily_stats: owner/manager select
bento_orders: owner/manager/front_desk full; kitchen select; kitchen status only by RPC
bento_weekly_menu: owner/manager write; kitchen/front_desk select
bento_customers: owner/manager/front_desk full
bento_subscription_days: owner/manager/front_desk full
bento_holidays: owner/manager write; kitchen/front_desk select
purchase_items: owner/manager/kitchen full
tasks: all roles full
incidents: owner/manager/front_desk full
staff_profiles: self select; owner full; manager may read display fields through a restricted view
staff_sessions: self select through a restricted view; start/touch/end only by RPC; owner select/update
audit_logs: owner select; no client update/delete
```

Use dynamic `to_regclass()` checks for optional tables so migration order does not fail if a placeholder business table is not yet present.

Explicitly revoke business-table access from `anon`. Grant only the minimum table/function privileges needed by `authenticated`; RLS then applies the staff-role and valid-session checks. Remove or replace any pre-existing permissive anonymous policies discovered during staging review.
