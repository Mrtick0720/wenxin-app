# Wenxin Assets Module — Architecture Specification

**Date:** 2026-06-08
**Status:** Draft, pending review
**Depends on:** Staff Authentication (approved, implemented)
**Product Spec:** [assets-module.md](assets-module.md)

## 1. Architecture Overview

The Assets module is a **standalone L2 module** under the Assets & Cash category. It adds one database table, one new L2 route (`/assets`), extends the existing `incidents` and `tasks` tables with `asset_id` foreign keys, and provides a lightweight asset registry with status lifecycle, warranty tracking, and incident/task linkage.

All new tables use Row Level Security matching the four-role model. All asset mutations are audit-logged via the existing `write_operational_audit()` trigger function. The table includes `outlet_id` for future multi-outlet support.

### Module Hierarchy

```
Assets & Cash
  ├── Assets    (/assets)   ← NEW
  └── Cashier   (/cashier, future)
```

### Principles

- **Asset registry, not maintenance system.** V1 is an equipment directory. Maintenance is handled through linked Incidents and Tasks — there is no separate maintenance module, scheduler, or work order system.
- **Warranty status is derived.** Expired, expiring soon, and active are computed from `warranty_expiry` at query time. Never stored.
- **Incident/task linkage is optional and manual.** Staff create an incident and optionally link it to an asset. No automatic incident generation from asset status changes.
- **Server actions for mutations.** Asset CRUD and status changes go through server actions to ensure audit logging and validation.
- **Existing patterns.** Route protection uses `requireRole()` / `proxy.ts`. Data access uses `createServerSupabaseClient()` for server components and `supabase` from `@/lib/supabase/client` for client components. Audit logging reuses the existing `write_operational_audit()` trigger.
- **Multi-outlet ready.** The `assets` table includes `outlet_id` with a default.

## 2. Data Model

### 2.1 `assets`

Core asset registry. One row per asset per outlet.

```sql
create table public.assets (
  id bigint generated by default as identity primary key,
  outlet_id uuid not null references public.outlets(id) on delete cascade
    default '00000000-0000-0000-0000-000000000001',
  asset_code text not null,
  name text not null,
  category text not null check (category in (
    'pos', 'printer', 'kitchen_equipment', 'refrigeration',
    'networking', 'furniture', 'other'
  )),
  description text,
  serial_number text,
  location text,
  purchase_date date,
  purchase_price numeric(10,2),
  warranty_expiry date,
  status text not null default 'active' check (status in (
    'active', 'under_repair', 'retired', 'disposed'
  )),
  notes text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (outlet_id, asset_code),
  unique (outlet_id, name)
);

create index assets_status_idx on public.assets(status, outlet_id);
create index assets_category_idx on public.assets(category, outlet_id);
create index assets_warranty_idx on public.assets(warranty_expiry)
  where warranty_expiry is not null;
```

**Column notes:**
- `asset_code` is unique per outlet. This is the internal identifier used for physical labeling (e.g., `FRZ-01`, `PRN-02`). Required.
- `name` is unique per outlet. Human-readable name (e.g., "Kitchen Freezer #1"). Required.
- `category` is an enum. Extensible via migration. Used for filtering.
- `serial_number` is for manufacturer identification. Not unique (different manufacturers may use overlapping serials).
- `location` is free-text (e.g., "Kitchen", "Counter", "Office"). Not an enum — locations vary per restaurant layout.
- `purchase_date` and `purchase_price` are for reference. Not used in calculations in v1.
- `warranty_expiry` is nullable — not all assets have warranties. The partial index supports efficient warranty-expiry queries.
- `status` is an enum with the four lifecycle states. `disposed` is terminal.

### 2.2 Extension to `incidents`

Add a nullable FK to link incidents to assets:

```sql
alter table public.incidents
  add column if not exists asset_id bigint
  references public.assets(id) on delete set null;

create index if not exists incidents_asset_idx
  on public.incidents(asset_id);
```

### 2.3 Extension to `tasks`

Add a nullable FK to link tasks to assets:

```sql
alter table public.tasks
  add column if not exists asset_id bigint
  references public.assets(id) on delete set null;

create index if not exists tasks_asset_idx
  on public.tasks(asset_id);
```

**Notes:**
- `on delete set null` — deleting an asset preserves linked incidents and tasks with `asset_id = null`.
- Both columns are nullable. Most incidents and tasks will not reference an asset.
- No data migration is needed. Existing rows have `asset_id = null`.
- The existing `incidents` and `tasks` tables already have `checklist_instance_id` from the Checklist module migration. Adding `asset_id` follows the same pattern.

## 3. Derived Values

### 3.1 Warranty Status

Computed at query time from `warranty_expiry`. Never stored.

```typescript
type WarrantyStatus = 'active' | 'expiring_soon' | 'expired'

function getWarrantyStatus(
  warrantyExpiry: string | null,
  today: Date = new Date()
): WarrantyStatus | null {
  if (!warrantyExpiry) return null
  const expiry = new Date(warrantyExpiry)
  const thirtyDaysFromNow = new Date(today)
  thirtyDaysFromNow.setDate(thirtyDaysFromNow.getDate() + 30)

  if (expiry < today) return 'expired'
  if (expiry <= thirtyDaysFromNow) return 'expiring_soon'
  return 'active'
}
```

### 3.2 Linked Incidents and Tasks

Queried from the `incidents` and `tasks` tables in the asset detail server component:

```typescript
async function getLinkedIncidents(
  supabase: SupabaseClient,
  assetId: number
): Promise<Incident[]> {
  const { data } = await supabase
    .from('incidents')
    .select('*')
    .eq('asset_id', assetId)
    .order('date', { ascending: false })
    .order('id', { ascending: false })
  return (data || []) as Incident[]
}

async function getLinkedTasks(
  supabase: SupabaseClient,
  assetId: number
): Promise<Task[]> {
  const { data } = await supabase
    .from('tasks')
    .select('*')
    .eq('asset_id', assetId)
    .order('date', { ascending: false })
    .order('id', { ascending: false })
  return (data || []) as Task[]
}
```

## 6. RLS Policies

### 6.1 `assets`

```sql
-- All authenticated: read all assets
create policy assets_read on public.assets
  for select to authenticated
  using (true);

-- Owner and Manager: manage assets
create policy assets_manage on public.assets
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));
```

All staff can read assets. Kitchen and Front Desk need visibility to identify equipment when reporting issues. Write access is restricted to Owner and Manager.

### 6.2 `incidents` and `tasks` (Existing — Add `asset_id` Write)

The existing RLS policies on `incidents` and `tasks` already control who can insert and update. The `asset_id` column inherits this protection. When creating an incident, staff with incident-creation permission can optionally set `asset_id`. No additional policy is needed.

## 7. Audit Logging

### 7.1 Trigger Attachment

```sql
create trigger write_operational_audit
  after insert or update or delete on public.assets
  for each row execute function public.write_operational_audit();
```

### 7.2 What Gets Logged

| Event | entity_type | Captured |
|-------|-------------|----------|
| Asset created | `assets` | asset_code, name, category, description, serial_number, location, purchase_date, purchase_price, warranty_expiry, status |
| Asset edited | `assets` | before/after of changed fields |
| Status changed | `assets` | before (old status), after (new status) |
| Asset deleted | `assets` | the deleted asset row |

No passwords, tokens, or secrets are stored in the `assets` table. The existing `PRIVATE_KEY_PATTERN` sanitizer is sufficient.

### 7.3 Incident/Task Linkage Auditing

When an incident or task is linked to an asset (via `asset_id`), the audit entry is captured by the existing `write_operational_audit` trigger on the `incidents` and `tasks` tables. No additional audit configuration is needed.

### 8.3 Server Actions

```
app/assets/actions.ts — new file

createAssetAction(formData)
  ├── requireRole('owner', 'manager')
  ├── Validate: asset_code is non-empty, asset_code is unique per outlet,
  │         name is non-empty, name is unique per outlet, category is valid
  ├── INSERT into assets (asset_code, name, category, description, serial_number, location,
  │         purchase_date, purchase_price, warranty_expiry, notes)
  ├── Audit: logged by trigger
  ├── revalidatePath('/assets')
  └── Return { success, assetId }

updateAssetAction(assetId, updates)
  ├── requireRole('owner', 'manager')
  ├── Validate: if asset_code or name changed, check uniqueness
  ├── If status is 'disposed': return error "Disposed assets cannot be modified"
  ├── UPDATE assets SET ... WHERE id = assetId
  ├── Audit: logged by trigger
  └── revalidatePath('/assets/[id]')

changeAssetStatusAction(assetId, newStatus)
  ├── requireRole('owner', 'manager')
  ├── Validate: newStatus is valid enum value
  ├── If current status is 'disposed': return error "Disposed assets cannot be modified"
  ├── UPDATE assets SET status = newStatus WHERE id = assetId
  ├── Audit: logged by trigger
  └── revalidatePath('/assets/[id]')

deleteAssetAction(assetId)
  ├── requireRole('owner')
  ├── SELECT count of linked incidents and tasks (for confirmation message)
  ├── DELETE from assets WHERE id = assetId
  │     (linked incidents/tasks have asset_id set to null via ON DELETE SET NULL)
  ├── Audit: logged by trigger
  └── revalidatePath('/assets')
```

## 9. Integration Wiring

### 9.1 Incidents Integration

**Schema:** `incidents.asset_id` (nullable FK → `assets.id ON DELETE SET NULL`).

**Usage:**
- When creating an incident, the incident form includes an optional "Linked Asset" dropdown populated from `assets WHERE status != 'disposed' ORDER BY name`.
- The asset detail view queries `incidents WHERE asset_id = X` to show linked incidents.
- The incident detail view shows the linked asset name as a tappable link (if `asset_id` is set).

**No automatic generation.** Unlike the Checklist module, asset status changes do not auto-create incidents. Staff manually create an incident and link it.

### 9.2 Tasks Integration

**Schema:** `tasks.asset_id` (nullable FK → `assets.id ON DELETE SET NULL`).

**Usage:**
- When creating a task, the task form includes an optional "Linked Asset" dropdown.
- The asset detail view queries `tasks WHERE asset_id = X` to show linked tasks.

### 9.3 Checklist Integration (Future)

When the Checklist module supports asset references on template items, a future migration adds `asset_id` to `checklist_template_items` or `checklist_item_responses`. The asset detail view can then show checklist history. This is deferred.

### 9.4 Home Dashboard Integration

**Quick Access card:**

```typescript
{ href: '/assets', label: 'Assets' }
```

Visible to all roles.

**Today's Issues:**
Assets with status `under_repair` appear as informational items:
```typescript
{ type: 'Asset Under Repair', detail: asset.name, link: '/assets' }
```

### 9.5 Incident Form Enhancement (Future)

The Incidents module can be enhanced to include an "Asset" dropdown when `incident_type` is `equipment`. This is an Incidents module enhancement, not an Assets module deliverable. The FK column is ready.

## 11. Summary of Constraints

1. **Standalone L2 module at `/assets`.** Registered in `stackRoutes.tsx` as a lazy-loaded client component.
2. **No new BottomNav tab.** Accessed from Quick Access grid on Home.
3. **No dashboard redesign.** Home dashboard gains one Quick Access card. Assets under repair appear in Today's Issues.
4. **All staff can view assets.** Kitchen and Front Desk need equipment visibility to report issues. Only Owner and Manager can create/edit.
5. **No automatic incident generation.** Unlike the Checklist module, asset status changes do not auto-create incidents. Staff manually link incidents to assets.
6. **No Maintenance module.** Equipment issues are tracked through linked Incidents and Tasks. Maintenance scheduling is deferred to v2.
7. **`incidents` and `tasks` are extended with nullable FKs.** Same pattern as `checklist_instance_id`. No data migration. Existing rows unaffected.
8. **Warranty status is derived, not stored.** Expired/expiring/active is computed from `warranty_expiry` at query time.
9. **`disposed` is a terminal status.** Disposed assets cannot be modified. Deletion unlinks incidents and tasks via `ON DELETE SET NULL`.
10. **Multi-outlet ready.** The `assets` table includes `outlet_id`. The `outlets` table is created with `if not exists`.
11. **No changes to existing auth, RLS, or audit infrastructure.** The module consumes these services.
12. **The existing `write_operational_audit()` trigger is reused.**
13. **All visible text is English.**
