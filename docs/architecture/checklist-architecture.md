# Wenxin Checklist Module — Architecture Specification

**Date:** 2026-06-08
**Status:** Draft, pending review
**Depends on:** Staff Authentication (approved, implemented)
**Product Spec:** [checklist-module.md](checklist-module.md)

## 1. Architecture Overview

The Checklist module is a **standalone L2 module** under the Operations Control category. It adds four database tables, one new L2 route (`/checklist`), auto-scheduling logic for daily recurring checklists, item-by-item completion with incident/task generation, and supervisor verification workflow. It integrates with Attendance, Cashier (future), Inventory (future), Incidents, and Tasks modules.

All new tables use Row Level Security matching the four-role model. All checklist actions are audit-logged via the existing `write_operational_audit()` trigger function. All tables include `outlet_id` for future multi-outlet support.

### Module Hierarchy

```
Operations Control
  ├── Complaints   (/complaints)
  ├── Incidents    (/incidents)
  ├── Tasks        (/tasks)
  └── Checklist    (/checklist)  ← NEW
```

### Principles

- **Completeness is derived.** An instance is `completed` when all item responses are non-pending. This is checked at query time and after each response save.
- **Overdue is derived.** An instance is overdue when `scheduled_time < now` and status is `pending` or `in_progress`. Never stored.
- **Server actions for mutations.** Item responses, verification, and template edits go through server actions to ensure audit logging, incident/task generation, and attendance gating are consistent.
- **Existing patterns.** Route protection uses `requireRole()` / `proxy.ts`. Data access uses `createServerSupabaseClient()` for server components and `supabase` from `@/lib/supabase/client` for client components. Audit logging reuses the existing `write_operational_audit()` trigger.
- **Multi-outlet ready.** All tables have `outlet_id` with a default. The current single-outlet app uses a hardcoded default outlet UUID. Multi-outlet filtering is deferred to the UI layer.

## 2. Data Model

### 2.1 `outlets`

A minimal seed table for future multi-outlet support. In v1, a single default outlet row exists. All checklist tables reference it.

```sql
create table public.outlets (
  id uuid primary key default gen_random_uuid(),
  name text not null default 'Wenxin Main',
  created_at timestamptz not null default now()
);

-- Seed the default outlet
insert into public.outlets (id, name) values (
  '00000000-0000-0000-0000-000000000001',
  'Wenxin Main'
);
```

### 2.2 `checklist_templates`

Defines a checklist type, its items, and scheduling rules. Seeded with five templates in the migration.

```sql
create table public.checklist_templates (
  id bigint generated by default as identity primary key,
  outlet_id uuid not null references public.outlets(id) on delete cascade
    default '00000000-0000-0000-0000-000000000001',
  name text not null,
  checklist_type text not null check (checklist_type in (
    'opening', 'closing', 'kitchen_hygiene', 'stock_check', 'cash_closing'
  )),
  description text not null default '',
  assigned_role text not null check (assigned_role in (
    'owner', 'manager', 'kitchen', 'front_desk'
  )),
  is_recurring boolean not null default true,
  requires_verification boolean not null default true,
  is_active boolean not null default true,
  sort_order integer not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (outlet_id, checklist_type)
);

create index checklist_templates_type_idx on public.checklist_templates(checklist_type);
```

**Column notes:**
- `checklist_type` is an enum. Only one template per type per outlet (unique constraint). This means there is exactly one Opening Checklist template, one Closing Checklist template, etc.
- `assigned_role` determines which role is responsible. This is a single role per template in v1.
- `is_active` controls whether instances are auto-created. Owner can deactivate a template to pause it.
- Scheduling is defined on `checklist_template_runs`, not on the template directly. This allows templates to have multiple scheduled runs per day (e.g., Kitchen Hygiene at 10:00 and 16:00).

### 2.3 `checklist_template_runs`

Defines when a checklist template runs. Each row is one scheduled run. A template with two runs per day (e.g., Kitchen Hygiene) has two rows.

```sql
create table public.checklist_template_runs (
  id bigint generated by default as identity primary key,
  template_id bigint not null references public.checklist_templates(id) on delete cascade,
  run_key text not null,
  scheduled_time time not null,
  created_at timestamptz not null default now(),
  unique (template_id, run_key)
);

create index checklist_template_runs_template_idx
  on public.checklist_template_runs(template_id);
```

**Seeded runs:**

| Template | run_key | scheduled_time |
|----------|---------|---------------|
| Opening Checklist | `opening` | 08:00 |
| Closing Checklist | `closing` | 21:00 |
| Kitchen Hygiene | `morning` | 10:00 |
| Kitchen Hygiene | `afternoon` | 16:00 |
| Stock Check | `daily` | 09:00 |
| Cash Closing | `cash_closing` | 21:00 |

**Column notes:**
- `run_key` is a stable identifier for this run within its template. It is used in the uniqueness constraint on instances: `(template_id, business_date, outlet_id, run_key)`.
- `scheduled_time` is a `time` (not `timestamptz`) — the time of day when this run should be scheduled. Combined with `business_date` at instance creation time.
- The unique constraint on `(template_id, run_key)` ensures no duplicate run keys within a template.
- In v1, runs are seeded and not editable from the UI. In v2, Owner/Manager can add, remove, or reschedule runs.

### 2.4 `checklist_template_items`

Ordered items within a template. Each item is a specific check to perform.

```sql
create table public.checklist_template_items (
  id bigint generated by default as identity primary key,
  template_id bigint not null references public.checklist_templates(id) on delete cascade,
  description text not null,
  category text not null default 'General',
  sort_order integer not null default 0,
  is_critical boolean not null default false,
  requires_note_on_fail boolean not null default true,
  inventory_ref text,  -- optional link to inventory item SKU/category
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index checklist_template_items_template_idx
  on public.checklist_template_items(template_id, sort_order);
```

**Column notes:**
- `inventory_ref` is a free-text reference for Stock Check items. When the Inventory module is built with real data, this becomes a foreign key. For v1, it is a tag used by the alert generation logic (e.g., `"cooking_oil"`, `"soy_sauce"`).
- `is_critical` determines whether a failure creates an incident (true) or a task (false).
- `requires_note_on_fail` is always true for critical items. For non-critical items, it can be set to false (a fail without explanation is allowed).

### 2.5 `checklist_instances`

A specific run of a checklist on a given business date. Created by the auto-scheduling engine. One instance per `(template_id, business_date, outlet_id, run_key)`.

```sql
create table public.checklist_instances (
  id bigint generated by default as identity primary key,
  template_id bigint not null references public.checklist_templates(id) on delete cascade,
  outlet_id uuid not null references public.outlets(id) on delete cascade
    default '00000000-0000-0000-0000-000000000001',
  run_key text not null,
  business_date date not null,
  scheduled_time timestamptz not null,
  status text not null default 'pending' check (status in (
    'pending', 'in_progress', 'completed', 'verified'
  )),
  assigned_role text not null,
  completed_by uuid references public.staff_profiles(id),
  started_at timestamptz,
  completed_at timestamptz,
  verified_by uuid references public.staff_profiles(id),
  verified_at timestamptz,
  verification_note text,
  rejection_note text,
  notes text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index checklist_instances_date_idx on public.checklist_instances(business_date);
create index checklist_instances_status_idx on public.checklist_instances(status, business_date);
create index checklist_instances_template_date_idx
  on public.checklist_instances(template_id, business_date);
create unique index checklist_instances_one_active_per_run
  on public.checklist_instances(template_id, business_date, outlet_id, run_key)
  where status in ('pending', 'in_progress');
```

**Column notes:**
- `run_key` identifies which run this instance belongs to (e.g., `morning`, `afternoon`, `opening`). Copied from `checklist_template_runs` at creation time.
- `scheduled_time` is a `timestamptz` — the full timestamp derived from `checklist_template_runs.scheduled_time` (time-only) combined with `business_date`.
- `status` does not include `overdue`. Overdue is derived: `scheduled_time < now() AND status IN ('pending', 'in_progress')`.
- `assigned_role` is copied from the template at instance creation. Denormalized for query convenience and to preserve the assignment even if the template is later edited.
- The partial unique index `checklist_instances_one_active_per_run` enforces BR-8: only one active instance per run per business date. Templates with multiple runs (Kitchen Hygiene: `morning` + `afternoon`) each get their own instance. No special-case code.
- `rejection_note` is set when a Manager rejects a completed checklist (status returns to `in_progress`).
- `verification_note` is set when a Manager approves.

### 2.6 `checklist_item_responses`

Individual responses to template items within a checklist instance.

```sql
create table public.checklist_item_responses (
  id bigint generated by default as identity primary key,
  instance_id bigint not null references public.checklist_instances(id) on delete cascade,
  template_item_id bigint not null references public.checklist_template_items(id) on delete cascade,
  status text not null default 'pending' check (status in (
    'pending', 'pass', 'fail', 'skip'
  )),
  note text,
  responded_by uuid references public.staff_profiles(id),
  responded_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (instance_id, template_item_id)
);

create index checklist_item_responses_instance_idx
  on public.checklist_item_responses(instance_id);
```

**Column notes:**
- The unique constraint on `(instance_id, template_item_id)` enforces one response per item per instance.
- `responded_by` records who tapped the Pass/Fail/Skip button. May differ from `completed_by` on the instance if a Manager corrects a response.
- `note` is required when `status IN ('fail', 'skip')`. Enforced at the application layer, not by a database constraint (notes are free text).

### 2.7 Auto-Scheduling Logic

Checklist instances are created by a database function. It iterates over template runs and creates one instance per run per day. For v1, lazy scheduling is used: instances are created when the first staff member accesses the Checklist module each day.

```sql
create or replace function public.schedule_daily_checklists(target_date date)
returns setof public.checklist_instances
language plpgsql
security definer
set search_path = public
as $$
declare
  run_record record;
  instance public.checklist_instances;
  existing_count integer;
  scheduled_ts timestamptz;
begin
  for run_record in
    select
      r.id as run_id,
      r.run_key,
      r.scheduled_time,
      t.id as template_id,
      t.assigned_role
    from public.checklist_template_runs r
    join public.checklist_templates t on t.id = r.template_id
    where t.is_active and t.is_recurring
  loop
    -- Check if an active instance already exists for this run today
    select count(*) into existing_count
    from public.checklist_instances
    where template_id = run_record.template_id
      and business_date = target_date
      and run_key = run_record.run_key
      and status in ('pending', 'in_progress');

    if existing_count = 0 then
      scheduled_ts := target_date + run_record.scheduled_time;

      insert into public.checklist_instances (
        template_id, run_key, business_date, scheduled_time,
        assigned_role
      ) values (
        run_record.template_id, run_record.run_key, target_date, scheduled_ts,
        run_record.assigned_role
      )
      returning * into instance;

      -- Create pending item responses for every template item
      insert into public.checklist_item_responses (
        instance_id, template_item_id, status
      )
      select instance.id, ti.id, 'pending'
      from public.checklist_template_items ti
      where ti.template_id = run_record.template_id
      order by ti.sort_order;

      return next instance;
    end if;
  end loop;
end;
$$;
```

**When this runs:**
- The function is called when the first staff member of the day accesses the Checklist module (lazy scheduling). This avoids dependency on Supabase cron.
- It iterates over all active recurring template runs and creates one instance per run per business date.
- Kitchen Hygiene naturally gets two instances (`morning` + `afternoon`) because it has two rows in `checklist_template_runs`. No special-case code.
- The per-run uniqueness is enforced by both the application (the `existing_count` check) and the database (the partial unique index `checklist_instances_one_active_per_run`).

## 3. Status Derivation

### 3.1 Instance Completion

```typescript
function isInstanceComplete(
  responses: ChecklistItemResponse[]
): boolean {
  return responses.every(r => r.status !== 'pending')
}
```

Checked after each item response save. If all items are non-pending, the instance transitions to `completed`.

### 3.2 Instance Overdue

```typescript
function isOverdue(instance: ChecklistInstance): boolean {
  if (!['pending', 'in_progress'].includes(instance.status)) return false
  return new Date() > new Date(instance.scheduled_time)
}
```

### 3.3 Compliance Score

```typescript
function complianceScore(responses: ChecklistItemResponse[]): number {
  const total = responses.length
  if (total === 0) return 0
  const passed = responses.filter(r => r.status === 'pass').length
  return Math.round((passed / total) * 100)
}
```

## 6. RLS Policies

### 6.1 `checklist_templates`

```sql
-- All authenticated: read active templates
create policy checklist_templates_read on public.checklist_templates
  for select to authenticated
  using (is_active or public.staff_role_is(array['owner', 'manager']));

-- Owner and Manager: manage templates
create policy checklist_templates_manage on public.checklist_templates
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));

-- Owner only: delete templates
create policy checklist_templates_delete on public.checklist_templates
  for delete to authenticated
  using (public.staff_role_is(array['owner']));
```

### 6.2 `checklist_template_items`

```sql
-- Inherits access through template: read if you can read the parent template
create policy checklist_template_items_read on public.checklist_template_items
  for select to authenticated
  using (
    exists (
      select 1 from public.checklist_templates t
      where t.id = template_id
        and (t.is_active or public.staff_role_is(array['owner', 'manager']))
    )
  );

-- Owner and Manager: manage items
create policy checklist_template_items_manage on public.checklist_template_items
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));
```

### 6.3 `checklist_instances`

```sql
-- Owner and Manager: read all instances
create policy checklist_instances_read_all on public.checklist_instances
  for select to authenticated
  using (public.staff_role_is(array['owner', 'manager']));

-- Staff: read instances assigned to their role
create policy checklist_instances_read_role on public.checklist_instances
  for select to authenticated
  using (
    assigned_role = (
      select role from public.staff_profiles where id = auth.uid()
    )
  );

-- Staff: update instances assigned to their role (start, complete)
create policy checklist_instances_update_role on public.checklist_instances
  for update to authenticated
  using (
    assigned_role = (
      select role from public.staff_profiles where id = auth.uid()
    )
    and completed_by is null  -- not yet claimed by another staff member
  )
  with check (
    assigned_role = (
      select role from public.staff_profiles where id = auth.uid()
    )
  );

-- Owner and Manager: update any instance (verification, correction)
create policy checklist_instances_manage on public.checklist_instances
  for update to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));

-- System: insert instances (via auto-schedule function)
create policy checklist_instances_insert_system on public.checklist_instances
  for insert to authenticated
  with check (true);  -- the scheduling function is security definer
```

### 6.4 `checklist_item_responses`

```sql
-- Read: inherit from parent instance visibility
create policy checklist_item_responses_read on public.checklist_item_responses
  for select to authenticated
  using (
    exists (
      select 1 from public.checklist_instances i
      where i.id = instance_id
        and (
          public.staff_role_is(array['owner', 'manager'])
          or i.assigned_role = (
            select role from public.staff_profiles where id = auth.uid()
          )
        )
    )
  );

-- Staff: insert/update responses for assigned instances
create policy checklist_item_responses_respond on public.checklist_item_responses
  for all to authenticated
  using (
    exists (
      select 1 from public.checklist_instances i
      where i.id = instance_id
        and i.assigned_role = (
          select role from public.staff_profiles where id = auth.uid()
        )
        and i.completed_by is null
    )
  )
  with check (
    exists (
      select 1 from public.checklist_instances i
      where i.id = instance_id
        and i.assigned_role = (
          select role from public.staff_profiles where id = auth.uid()
        )
    )
  );

-- Owner and Manager: full access
create policy checklist_item_responses_manage on public.checklist_item_responses
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));
```

```

### 8.3 Server Actions

```
app/checklist/actions.ts — new

respondToItemAction(instanceId, templateItemId, status, note?)
  ├── requireCurrentStaff()
  ├── Validate: open attendance session exists
  ├── Validate: instance assigned_role matches staff role OR staff is owner/manager
  ├── Validate: instance status is 'pending' or 'in_progress'
  ├── If instance status is 'pending':
  │     ├── UPDATE instance SET status = 'in_progress', started_at = now(), completed_by = uid
  ├── UPSERT checklist_item_responses (status, note, responded_by, responded_at)
  ├── Check if all items responded → if yes, auto-complete (see completeChecklistAction)
  └── revalidatePath('/checklist/[id]')

completeChecklistAction(instanceId)
  ├── requireCurrentStaff()
  ├── Validate: all item responses are non-pending
  ├── UPDATE instance SET status = 'completed', completed_at = now()
  ├── For each failed critical item → INSERT incident
  ├── For each failed non-critical item → INSERT task
  ├── For each failed Stock Check item with inventory_ref → INSERT inventory_alert
  ├── Audit: logged by trigger

### 9.2 Incident Generation

**In `completeChecklistAction`:**

```typescript
for (const response of failedResponses) {
  const item = templateItems.find(ti => ti.id === response.template_item_id)
  if (!item?.is_critical) continue

  // Prevent duplicates
  const { data: existing } = await supabase
    .from('incidents')
    .select('id')
    .eq('checklist_instance_id', instanceId)
    .eq('title', `Checklist Failure: ${template.name} — ${item.description}`)
    .maybeSingle()

  if (!existing) {
    await supabase.from('incidents').insert({
      date: businessDate,
      title: `Checklist Failure: ${template.name} — ${item.description}`,
      incident_type: 'checklist',
      severity: 'high',
      status: 'open',
      checklist_instance_id: instanceId,
      reported_by: instance.completed_by,
    })
  }
}
```

**Note:** This requires adding `checklist_instance_id` and `reported_by` columns to the `incidents` table. If these columns do not already exist, the migration adds them. The `incidents` table is already in production with RLS and audit triggers — columns are added with `add column if not exists`.

### 9.5 Inventory Alert Generation

**In `completeChecklistAction`:**

```typescript
for (const response of failedResponses) {
  const item = templateItems.find(ti => ti.id === response.template_item_id)
  if (!item?.inventory_ref) continue  // only Stock Check items with inventory refs

  await supabase.from('inventory_alerts').insert({
    business_date: businessDate,
    inventory_ref: item.inventory_ref,
    alert_type: 'checklist_fail',
    message: `Stock Check: ${item.description} — ${response.note || 'Failed'}`,
    checklist_instance_id: instanceId,
    status: 'open',
  })
}
```

**This requires an `inventory_alerts` table:**

```sql
create table if not exists public.inventory_alerts (
  id bigint generated by default as identity primary key,
  business_date date not null default current_date,
  inventory_ref text not null,
  alert_type text not null default 'checklist_fail',
  message text not null,
  checklist_instance_id bigint references public.checklist_instances(id) on delete set null,
  status text not null default 'open' check (status in ('open', 'acknowledged', 'resolved')),
  created_at timestamptz not null default now()
);
```

This table is lightweight — it holds alert records until the Inventory module is built with real data. When Inventory is implemented, it queries this table or the alerts are migrated to the Inventory module's native data model.

## 11. Summary of Constraints

10. **The existing `write_operational_audit()` trigger is reused** on all four new tables.
