# Wenxin Suppliers Module — Architecture Specification

**Date:** 2026-06-08
**Status:** Draft, pending review
**Depends on:** Staff Authentication (approved, implemented)
**Product Spec:** [suppliers-module.md](suppliers-module.md)

## 1. Architecture Overview

The Suppliers module is a **standalone L2 module** under the Inventory & Procurement category. It adds two database tables, one new L2 route (`/suppliers`), extends the existing `purchase_items` table with a `supplier_id` foreign key, and provides supplier CRUD with contacts and purchase history integration.

All new tables use Row Level Security matching the four-role model. All supplier mutations are audit-logged via the existing `write_operational_audit()` trigger function. All tables include `outlet_id` for future multi-outlet support.

### Module Hierarchy

```
Inventory & Procurement
  ├── Inventory   (/inventory)
  ├── Purchase    (/purchase)
  └── Suppliers   (/suppliers)  ← NEW
```

### Principles

- **Supplier directory, not analytics.** V1 is a supplier information directory. Purchase history is displayed for reference. Performance analytics (on-time delivery, spend trends, rankings) are deferred.
- **Soft-delete only in v1.** Status changes (active → inactive, active → suspended) replace hard deletes. Purchase history is preserved via `ON DELETE SET NULL`.
- **Server actions for mutations.** Supplier and contact edits go through server actions to ensure audit logging and validation.
- **Existing patterns.** Route protection uses `requireRole()` / `proxy.ts`. Data access uses `createServerSupabaseClient()` for server components and `supabase` from `@/lib/supabase/client` for client components. Audit logging reuses the existing `write_operational_audit()` trigger.
- **Multi-outlet ready.** All tables have `outlet_id` with a default. The current single-outlet app uses a hardcoded default outlet UUID.

## 2. Data Model

### 2.1 `suppliers`

Core supplier profile. One row per supplier per outlet.

```sql
create table public.suppliers (
  id bigint generated by default as identity primary key,
  outlet_id uuid not null references public.outlets(id) on delete cascade
    default '00000000-0000-0000-0000-000000000001',
  name text not null,
  contact_person text,
  phone text,
  whatsapp text,
  email text,
  address text,
  payment_terms text check (payment_terms in (
    'cod', 'net_15', 'net_30', 'net_60', 'other'
  )),
  status text not null default 'active' check (status in (
    'active', 'inactive', 'suspended'
  )),
  notes text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (outlet_id, name)
);

create index suppliers_status_idx on public.suppliers(status, outlet_id);
create index suppliers_name_idx on public.suppliers(name);
```

**Column notes:**
- `name` is unique per outlet. Supplier names must be unique within an outlet but can repeat across outlets in the future.
- `contact_person` is a denormalized copy of the primary contact's name for quick display in lists. Updated automatically when the primary contact changes.
- `payment_terms` is an enum of common terms. `other` allows free-text in the `notes` field.
- `status` drives visibility: `active` appears in dropdowns; `inactive` and `suspended` are hidden from defaults but searchable.
- The `outlets` table was created by the Checklist module migration. If the Suppliers migration runs first, it creates `outlets` with `if not exists`.

### 2.2 `supplier_contacts`

Contact persons at the supplier organization. Multiple per supplier.

```sql
create table public.supplier_contacts (
  id bigint generated by default as identity primary key,
  supplier_id bigint not null references public.suppliers(id) on delete cascade,
  name text not null,
  role text,
  phone text,
  email text,
  is_primary boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index supplier_contacts_supplier_idx
  on public.supplier_contacts(supplier_id);
create unique index supplier_contacts_one_primary
  on public.supplier_contacts(supplier_id)
  where is_primary = true;
```

**Column notes:**
- The partial unique index `supplier_contacts_one_primary` enforces BR-3: at most one primary contact per supplier. The application layer ensures at least one primary exists for Active suppliers.
- `on delete cascade` — deleting a supplier removes all its contacts. (Suppliers are soft-deleted in practice.)
- `role` is free-text (not an enum) to accommodate diverse supplier organizations.

### 2.3 Extension to `purchase_items`

Add a nullable FK to link purchase items to suppliers:

```sql
alter table public.purchase_items
  add column if not exists supplier_id bigint
  references public.suppliers(id) on delete set null;

create index if not exists purchase_items_supplier_idx
  on public.purchase_items(supplier_id);
```

**Notes:**
- `on delete set null` — if a supplier is hard-deleted (rare, Owner-only edge case), purchase items are preserved with `supplier_id = null` while the text `supplier` field retains the original name.
- The existing `supplier` text column is **not removed**. It remains as a fallback display value when `supplier_id` is null.
- The Purchase module's item entry form can set `supplier_id` when the user selects a supplier from a dropdown.
- Existing purchase items with `supplier_id = null` but `supplier` text populated are displayed in a "Legacy (unlinked)" section of the supplier detail. A future data migration can match and backfill.

## 3. Derived Values

### 3.1 Last Purchase Date

Derived from `purchase_items` for display in the supplier list.

```typescript
async function getLastPurchaseDate(
  supabase: SupabaseClient,
  supplierId: number
): Promise<string | null> {
  const { data } = await supabase
    .from('purchase_items')
    .select('date')
    .eq('supplier_id', supplierId)
    .order('date', { ascending: false })
    .limit(1)
    .maybeSingle()
  return data?.date ?? null
}
```

### 3.2 Recent Purchases

Purchase history displayed for reference on the supplier detail page. Limited to the 30 most recent items.

```typescript
async function getRecentPurchases(
  supabase: SupabaseClient,
  supplierId: number,
  limit = 30
): Promise<PurchaseItem[]> {
  const { data } = await supabase
    .from('purchase_items')
    .select('*')
    .eq('supplier_id', supplierId)
    .order('date', { ascending: false })
    .order('id', { ascending: false })
    .limit(limit)
  return (data || []) as PurchaseItem[]
}
```

No performance analytics (total spend, on-time delivery rate, order frequency, rankings) are computed in v1. These are deferred.

## 6. RLS Policies

### 6.1 `suppliers`

```sql
-- Owner, Manager, Kitchen: read all suppliers
create policy suppliers_read on public.suppliers
  for select to authenticated
  using (public.staff_role_is(array['owner', 'manager', 'kitchen']));

-- Owner and Manager: manage suppliers
create policy suppliers_manage on public.suppliers
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));
```

### 6.2 `supplier_contacts`

```sql
-- Inherits parent supplier visibility: read if you can read the supplier
create policy supplier_contacts_read on public.supplier_contacts
  for select to authenticated
  using (
    exists (
      select 1 from public.suppliers s
      where s.id = supplier_id
        and public.staff_role_is(array['owner', 'manager', 'kitchen'])
    )
  );

-- Owner and Manager: manage contacts
create policy supplier_contacts_manage on public.supplier_contacts
  for all to authenticated
  using (public.staff_role_is(array['owner', 'manager']))
  with check (public.staff_role_is(array['owner', 'manager']));
```

### 6.3 `purchase_items` (Existing — Add `supplier_id` Write)

The existing `purchase_items` RLS policy allows `owner`, `manager`, `kitchen` to insert and update. The `supplier_id` column inherits this protection — Kitchen can set `supplier_id` when creating a purchase item, which is correct (they are picking a supplier from the dropdown during purchase entry). No additional policy is needed.

## 7. Audit Logging

### 7.1 Trigger Attachment

```sql
-- suppliers
create trigger write_operational_audit
  after insert or update or delete on public.suppliers
  for each row execute function public.write_operational_audit();

-- supplier_contacts
create trigger write_operational_audit
  after insert or update or delete on public.supplier_contacts
  for each row execute function public.write_operational_audit();
```

### 7.2 What Gets Logged

| Event | entity_type | Captured |
|-------|-------------|----------|
| Supplier created | `suppliers` | name, contact_person, phone, email, address, payment_terms, status |
| Supplier edited | `suppliers` | before/after of changed fields |
| Supplier status changed | `suppliers` | before (old status), after (new status) |
| Contact added | `supplier_contacts` | name, role, phone, email, is_primary |
| Contact edited | `supplier_contacts` | before/after of changed fields |
| Contact removed | `supplier_contacts` | the deleted contact row |

No passwords, tokens, or secrets are stored in these tables. The existing `PRIVATE_KEY_PATTERN` sanitizer is sufficient.

### 8.3 Server Actions

```
app/suppliers/actions.ts — new file

createSupplierAction(formData)
  ├── requireRole('owner', 'manager')
  ├── Validate: name is non-empty, name is unique per outlet
  ├── INSERT into suppliers (name, contact_person, phone, email, address, payment_terms, notes)
  ├── Audit: logged by trigger
  ├── revalidatePath('/suppliers')
  └── Return { success, supplierId }

updateSupplierAction(supplierId, updates)
  ├── requireRole('owner', 'manager')
  ├── Validate: name uniqueness if name is being changed
  ├── UPDATE suppliers SET ... WHERE id = supplierId
  ├── Audit: logged by trigger
  └── revalidatePath('/suppliers/[id]')

changeSupplierStatusAction(supplierId, newStatus)
  ├── requireRole('owner', 'manager')
  ├── Validate: newStatus is valid enum value
  ├── UPDATE suppliers SET status = newStatus WHERE id = supplierId
  ├── Audit: logged by trigger
  └── revalidatePath('/suppliers/[id]')

addContactAction(supplierId, contact)
  ├── requireRole('owner', 'manager')
  ├── Validate: name is non-empty
  ├── If contact.is_primary: unset is_primary on all other contacts for this supplier
  ├── INSERT into supplier_contacts
  ├── If this is the first contact: auto-set is_primary = true
  ├── Update suppliers.contact_person if is_primary
  ├── Audit: logged by trigger
  └── revalidatePath('/suppliers/[id]')

updateContactAction(contactId, updates)
  ├── requireRole('owner', 'manager')
  ├── If updates.is_primary: unset is_primary on sibling contacts
  ├── UPDATE supplier_contacts
  ├── If is_primary changed: update suppliers.contact_person
  ├── Audit: logged by trigger
  └── revalidatePath('/suppliers/[id]')

removeContactAction(contactId)
  ├── requireRole('owner', 'manager')
  ├── If this is the primary contact: check if supplier has other contacts
  │     ├── If other contacts exist: prompt to promote one (or auto-promote the first)
  │     └── If no other contacts and supplier is Active: return error "Active supplier must have a primary contact"
  ├── DELETE from supplier_contacts
  ├── Audit: logged by trigger
  └── revalidatePath('/suppliers/[id]')
```

## 9. Integration Wiring

### 9.1 Purchase Module Integration

**Current state:** `purchase_items` has a free-text `supplier` column. No supplier table exists.

**After integration:**

1. `purchase_items.supplier_id` (nullable FK) is added via migration.
2. The Purchase module's item entry form (`PurchaseClient.tsx`) gains a supplier dropdown populated from `suppliers WHERE status = 'active' ORDER BY name`.
3. When the user selects a supplier, `supplier_id` is set on the new purchase item. The text `supplier` field is also populated (denormalized for backward compatibility).
4. In the supplier detail view, recent purchases from this supplier are queried by `supplier_id`.
5. For existing purchase items where `supplier_id` is null but `supplier` text exists, they appear in a "Legacy purchases (unlinked)" section.

**Supplier dropdown in Purchase module (conceptual):**

```typescript
// In PurchaseClient.tsx — supplier selection
const { data: suppliers } = await supabase
  .from('suppliers')
  .select('id, name')
  .eq('status', 'active')
  .order('name')

// When saving a purchase item:
await supabase.from('purchase_items').insert({
  ...itemData,
  supplier: selectedSupplier.name,     // text fallback
  supplier_id: selectedSupplier.id,    // FK reference
})
```

### 9.2 Finance Module Integration (Future)

When the Finance module is built:

1. Supplier payment terms (`payment_terms` column) inform accounts payable workflows.
2. Spend per supplier is derivable from `purchase_items` when needed (deferred to v2).
3. Supplier payment history would be tracked in a finance-specific table referencing `suppliers.id`.

This is deferred. No Finance module changes are needed in v1.

### 9.3 Home Dashboard Integration

**Quick Access card:**

```typescript
{ href: '/suppliers', label: 'Suppliers' }
```

Visible to Owner, Manager, and Kitchen. No count badge in v1 (active supplier count is a minor metric).

## 11. Summary of Constraints

1. **Standalone L2 module at `/suppliers`.** Registered in `stackRoutes.tsx` as a lazy-loaded client component.
2. **No new BottomNav tab.** Accessed from Quick Access grid on Home.
3. **No dashboard redesign.** Home dashboard gains one Quick Access card.
4. **Kitchen has view-only access.** They can see supplier information for Purchase reference but cannot create or edit.
5. **Front Desk has no access.** The route is not in their permission set. RLS denies reads.
6. **`purchase_items` is extended with a nullable FK.** The existing `supplier` text column is preserved. No data migration. Existing rows are unaffected.
7. **Soft-delete only.** Status changes replace hard deletes. Purchase history is preserved via `ON DELETE SET NULL`.
8. **V1 is a supplier directory, not an analytics system.** Purchase history is displayed for reference. Performance analytics (spend trends, on-time delivery, rankings) are deferred to v2.
9. **Multi-outlet ready.** All tables include `outlet_id`. The `outlets` table is created with `if not exists` (shared with the Checklist module).
10. **No changes to existing auth, RLS, or audit infrastructure.** The module consumes these services.
11. **The existing `write_operational_audit()` trigger is reused** on all new tables.
12. **All visible text is English.**
